## 修复前端API与后端Controller不匹配的bug

### 问题根因
前端API调用与后端Controller接口存在大量不匹配，主要3类问题：
1. **参数传递方式不对** — 前端用 `data`(JSON body) 但后端用 `@RequestParam`(query参数)
2. **前端调用后端不存在的接口** — 需改用后端已有接口
3. **前端函数未导出** — article.js 缺少 getAllArticles

### 修复内容

#### 1. order.js
- `updateOrderStatus`: `data: { status }` → `params: { status }` (匹配后端 @RequestParam)
- `cancelOrder`: 移除不存在的 `/order/{id}/cancel`，改为调用 `updateOrderStatus(id, 4)`
- `addLogistics`: `/order/{orderId}/logistics` → `POST /logistics`（匹配 LogisticsController）
- `updateLogistics`: `PUT /logistics/{id}` → `PUT /logistics/{id}/status`（匹配后端）
- `getOrderStatistics`: `/order/statistics` → `/statistics/orders/monthly`

#### 2. user.js
- `updateUserStatus`: `PUT /user/{id}/status` → `PUT /user/{id}`（使用通用更新接口）
- `updateUserRole`: `PUT /user/{id}/role` → `PUT /user/{id}`（使用通用更新接口）

#### 3. article.js
- **新增** `getAllArticles` 函数（ArticleManagement.vue引用但未导出）
- `keyword` 参数映射为后端的 `title` 参数
- `getFeaturedArticles`: `/article/featured` → `/article/page?status=1`

#### 4. notice.js
- `getAllNotices`: `keyword` 参数映射为后端的 `title` 参数

#### 5. carousel.js
- `updateCarouselSort`: `PUT /carousel/{id}/sort` → `PUT /carousel/{id}`（使用通用更新）

#### 6. CarouselManagement.vue
- `loadCarousels`: 正确处理分页返回数据结构 `res.data.records`

#### 7. OrderManagement.vue
- 搜索参数 `keyword` 改为 `id`（后端不支持keyword搜索）

## 第二轮bug修复：图片404、轮播图字段不匹配、公告字段不匹配

### Bug 1：商品/轮播图/库存管理页面图片404
- **根因**：后端图片资源映射为 `/api/img/**`，但管理页面直接用 `row.imageUrl`（值如`/img/xxx.png`）渲染，缺少 `/api` 前缀
- **修复**：在 ProductManagement.vue、StockManagement.vue、CarouselManagement.vue 中引入 `getImageUrl()` 工具函数处理图片URL

### Bug 2：轮播图管理排序无效 & 数据不显示
- **根因**：前端字段名与后端 `CarouselItem` 实体不匹配
  - 前端 `sort` → 后端实际是 `sortOrder`
  - 前端 `linkUrl` → 后端没有此字段
- **修复**：
  - CarouselManagement.vue：`sort` → `sortOrder`，`linkUrl` → `description`
  - carousel.js：`updateCarouselSort` 传 `{ sortOrder }` 替代 `{ sort }`

### Bug 3：公告管理类型/创建时间不显示 & 按钮报错
- **根因**：前端字段与后端 `Notice` 实体完全不匹配
  - 后端实体只有：`id, title, content, time, tags`
  - 前端错误使用：`type`(INFO/WARNING/URGENT)、`status`(1/0)、`createdAt`
  - content formatter 中 `row.content.length` 在 content 为空时报错
- **修复**：
  - NoticeManagement.vue：`type` → `tags`，`createdAt` → `time`，移除 `status` 列和 `handleStatusChange`
  - 表单：`type` select 改为 `tags` select（通知/警告/紧急/活动），移除状态单选
  - content formatter 加空值保护

## 第三轮bug修复：图片仍404 + 轮播图崩溃

### Bug 1：商品图片仍然404
- **根因**：上一轮虽然在 ProductManagement.vue 导入了 `getImageUrl` 并定义了 `getFullImageUrl` 方法，但**模板中的 `:src="row.imageUrl"` 没有修改成功**，仍然直接使用 `row.imageUrl` 而非 `getFullImageUrl(row.imageUrl)`
- **修复**：模板中 `:src="row.imageUrl"` → `:src="getFullImageUrl(row.imageUrl)"`

### Bug 2：轮播图管理页面崩溃（"应用程序遇到错误"）
- **根因**：CarouselManagement.vue 的 methods 中调用了 `getImageUrl(url)`，但 **import 语句丢失了**——上一轮只添加了方法调用，漏掉了 `import { getImageUrl } from '@/utils/image'`
- **修复**：补上 import 语句

### Bug 3：库存管理页面崩溃（"应用程序遇到错误"）
- **根因**：StockManagement.vue 模板中调用了 `getFullImageUrl(row.imageUrl)`，已 import 了 `getImageUrl`，但 **methods 中没有定义 `getFullImageUrl` 方法**
- **修复**：在 methods 中补上 `getFullImageUrl(url) { return getImageUrl(url) }`

## 第四轮深度修复：库存"操作记录"功能全链路修复

### 问题全景
点击商品的"操作记录"按钮后报错"加载操作记录失败"，经全面排查发现**前后端严重不匹配**，涉及3处重大缺陷：

### Bug 1（核心）：前端请求了后端不存在的接口
- **前端** `getStockHistory()` 请求 `GET /stock/product/{id}/history`
- **后端** StockController **根本没有这个接口**，只有 `/stock/in/list` 和 `/stock/out/list`
- **结果**：请求 404，axios 拦截器捕获 → "加载操作记录失败"
- **修复**：后端 StockController 新增 `@GetMapping("/product/{productId}/history")`，StockService 新增 `getProductStockHistory()` 方法——合并入库(StockIn)+出库(StockOut)记录为统一格式，按 createdAt 倒序排列并手动分页

### Bug 2：入库/出库表单字段与后端实体不匹配
- **前端表单**只提交 `{productId, quantity, operator(字符串), remark}`
- **后端 StockIn 实体必填**：`productId, quantity, unitPrice(BigDecimal), stockDate(Date), operatorId(Long)`
- **后端 StockOut 实体必填**：`productId, quantity, unitPrice(BigDecimal), type(Integer), operatorId(Long)`
- **修复**：入库表单补充 unitPrice/supplier/stockDate/operatorId；出库表单补充 unitPrice/type/operatorId；使用独立的 stockInForm/stockOutForm

### Bug 3：操作记录弹窗表格列与实际数据结构不匹配
- **前端期望字段**：`type='IN'/'OUT', stockBefore, stockAfter, operator(字符串)`
- **后端实际无** `stockBefore`/`stockAfter` 字段
- **修复**：移除 stockBefore/stockAfter 列，改为 unitPrice/totalPrice/status/operatorName，适配 history 接口返回

### Bug 4：User 实体字段名错误
- 后端 history 接口中 `user.getNickname()` 调用报错——User 实体的用户名字段是 `name` 不是 `nickname`
- **修复**：`getNickname()` → `getName()`
